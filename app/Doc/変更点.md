# 変更履歴

## 変更点1
- **ホーム画面のダッシュボード削除**
  - `HomeScreen.kt` から商品カウント表示（全商品・未取得・終売）を削除し、メインメニューのみのシンプルな構成に変更しました。
- **スキャン画面への手入力モード追加**
  - `ScanScreen.kt`（3モードスキャン）、`GroupScanScreen.kt`（グループ追加）、`OrderScanScreen.kt`（発注スキャン）に、キーボード入力への切替機能を追加しました。
  - **ハイブリッド入力仕様**:
    - デフォルト（HIDモード）: Bluetoothスキャナー用に非表示の入力欄が常時フォーカスされ、ソフトキーボードは表示されません。
    - 手入力モード: TopAppBarのキーボードアイコンから切り替え可能。数値入力用の `OutlinedTextField` を表示し、ソフトキーボードでの入力に対応します。
    - **不具合修正**: 物理キーボードや一部の入力環境で、JANコード入力後にエンターキーを押しても確定されない問題を修正しました（`onKeyEvent` によるENTERキー補足を追加）。
    - **不具合修正**: 発注スキャン画面で「完了」ボタンを押しても画面が遷移しない（無反応に見える）問題を修正しました。セッション完了後に自動的に前の画面（ホーム画面）に戻るようナビゲーション処理を追加しました。
    - **不具合修正**: 発注スキャン画面で商品をスキャンしても「スキャン数」が0のまま増加しない問題を修正しました。新規セッション開始時にもDB監視用カウンタが正しく起動するようにプログラムを修正しました。

## 変更点2：AI連携の包括的改善（Gemini/Perplexity両対応）

**対象ファイル**: `WebViewJsHelper.kt`, `AiFetchViewModel.kt`, `ScanViewModel.kt`, `AiFetchScreen.kt`, `ConfirmMode.kt`

### 🐛 バグ修正: AI画面が常にGeminiで開く問題の解決
- **原因**: `aiUrl` の初期値がGeminiに固定されており、Settingsの読み込み完了前にWebViewが描画されていた。
- **対策**: `aiUrl` を `null` 許容型にし、初期値を `null` に設定。設定読み込み完了までプログレスインジケーターを表示し、確定後に正しいURLでWebViewを開くよう修正。

### 🔧 修正: 重要エディタ（Quill/Lexical）への自動入力対応
- **背景**: Gemini（Quill）やPerplexity（Lexical）は、`innerText` の直接書き換えではエディタの状態が更新されず、送信ボタンが反応しない。
- **対策**: `WebViewJsHelper.kt` に `getInjectPromptJsForRichEditor()` を新設。`document.execCommand('insertText', ...)` を使用してペースト入力をシミュレートすることで、内部状態の更新と送信ボタンの有効化を確実にした。

### 🎯 セレクタの精緻化
- **Gemini**: 
  - 入力欄: `div.ql-editor`（従来の汎用指定から専用クラスへ）
  - 送信ボタン: `button[aria-label='プロンプトを送信']`
- **Perplexity**: 
  - 入力欄: `#ask-input`（Lexical editor固有のID）
  - 送信ボタン: `button[aria-label='送信']`

### 🔄 各画面への水平展開
- 「AI一括取得画面」および「確認モードのスキャン詳細」の両方で、上記の修正を同期。どちらの導線からでも一貫して正しいAIが開き、プロンプトが自動入力されるように改善。

## 変更点3：AI商品情報取得機能の不具合修正と安定化

**対象ファイル**: `ProductMasterDao.kt`, `WebViewJsHelper.kt`, `AiFetchViewModel.kt`

### 🔍 修正: 「未取得商品」判定ロジックの改善
- **問題**: すでに名称やメーカー名が入力されている商品が、内部フラグの関係で「未取得」としてリストに残り続けていた。
- **対策**: `ProductMasterDao.kt` のクエリを修正。`info_fetched` フラグが 0 かつ「商品名またはメーカー名が空」の商品のみを抽出するように変更し、手動入力済みの商品が重複してリストアップされるのを防いだ。

### 🚀 強化: プロンプト注入（貼り付け）スクリプトの刷新
- **問題**: AIサイト（Gemini/Perplexity）の構造変化やブラウザの仕様により、プロンプトが入力欄に貼り付けられない場合があった。
- **対策**: 
  - `execCommand` に失敗した場合のフォールバックとして、`innerText` への直接代入や各種イベント（input, change, blur, keyup）の強制発火処理を追加。
  - 最新のセレクタ候補（ARIAラベルや特定クラス名）を追加し、マッチング精度を向上。
  - スクリプト実行結果（成功/失敗）を ViewModel 側で受け取れるようにし、実行状況の可視性を高めた。

### ⚠️ 改善: エラーハンドリングとステータス表示
- **対策**: `AiFetchViewModel.kt` において、貼り付けに失敗した際に「入力欄への貼り付け失敗。AIサイトの構造が変わった可能性があります。」という詳細なエラーメッセージをステータスバーに表示するようにし、トラブルシューティングを容易にした。

## 変更点4：自動貼り付け・取り込み機能の再設計と安定化

**対象ファイル**: `AiFetchViewModel.kt`, `ScanViewModel.kt`, `自動貼り付け取り込みの仕組み.md` (新規)

### 📚 設計ドキュメントの作成
- **新規ファイル**: `app/Doc/自動貼り付け取り込みの仕組み.md` を作成。
- **内容**: ターゲット特定（フォールバックチェーン）、状態考慮型インジェクション（execCommand + イベント発火）、ポーリング監視、JSON 解析の 4 段階の仕組みを詳述。他機能へも転用可能な設計指針を明確化した。

### 🛡️ 強化: 待機とリトライ（再試行）の実装
- **問題**: WebView のページ読み込み完了前にスクリプトが実行され、要素が見つからず失敗するケースがあった。
- **対策**:
  - **ページロード待機**: 注入前に現在の URL が AI サイトのものであるかを確認し、最大10秒間待機する処理を追加。
  - **リトライループ**: プロンプトの注入および送信ボタンのクリックにおいて、失敗時に最大3回まで再試行するロジックを導入。

### 🔄 統一: ViewModel 間のロジック共通化
- **対策**: 「一括取得（AiFetchViewModel）」で導入した最新の安定化ロジック（待機・リトライ・成功判定）を、「単発取得（ScanViewModel）」側にも完全移植。アプリ全体で一貫した高い成功率を確保した。

## 変更点5：Perplexity (Lexical エディタ) への完全対応

**対象ファイル**: `WebViewJsHelper.kt`, `自動貼り付け取り込みの仕組み.md`

### 🎯 修正: Perplexity 固有の要素特定
- **背景**: Perplexity の入力欄は `id="ask-input"` を持つ Lexical エディタであり、標準的なセレクタでは特定が不安定な場合があった。
- **対策**: `#ask-input` および `data-lexical-editor="true"` を最優先セレクタとして設定し、確実にターゲットを補足するように修正。

### ⚡ 強化: Lexical 互換のインジェクション
- **問題**: モダンなエディタフレームワーク（Lexical）では、単純なテキスト代入では「未入力」と判定され、送信ボタンが有効にならない。
- **対策**: 
  - `InputEvent` を生成し、`beforeinput` イベントを `insertText` タイプで明示的に発火させる処理を追加。
  - これによりエディタ内部のデータモデルが強制的に更新され、Web サイト側に「正しく入力された」と認識させることに成功。

### 🖱️ 修正: 送信ボタンのクリック精度向上
- **対策**: ボタン内部に複雑な SVG 構造がある場合でも、`aria-label="送信"` を持つ `button` 要素を確実に特定し、クリックイベントを伝達するようにセレクタチェーンを調整。
