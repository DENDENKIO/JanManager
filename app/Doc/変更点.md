## 変更点2：AI連携の包括的改善（Gemini/Perplexity両対応）

**対象ファイル**: `WebViewJsHelper.kt`, `AiFetchViewModel.kt`, `ScanViewModel.kt`, `AiFetchScreen.kt`, `ConfirmMode.kt`

### 🐛 バグ修正: AI画面が常にGeminiで開く問題の解決
- **原因**: `aiUrl` の初期値がGeminiに固定されており、Settingsの読み込み完了前にWebViewが描画されていた。
- **対策**: `aiUrl` を `null` 許容型にし、初期値を `null` に設定。設定読み込み完了までプログレスインジケーターを表示し、確定後に正しいURLでWebViewを開くよう修正。

### 🔧 修正: 重要エディタ（Quill/Lexical）への自動入力対応
- **背景**: Gemini（Quill）やPerplexity（Lexical）は、`innerText` の直接書き換えではエディタの状態が更新されず、送信ボタンが反応しない。
- **対策**: `WebViewJsHelper.kt` に `getInjectPromptJsForRichEditor()` を新設。`document.execCommand('insertText', ...)` を使用してペースト入力をシミュレートすることで、内部状態の更新と送信ボタンの有効化を確実にした。

### 🎯 セレクタの精緻化
- **Gemini**: 
  - 入力欄: `div.ql-editor`（従来の汎用指定から専用クラスへ）
  - 送信ボタン: `button[aria-label='プロンプトを送信']`
- **Perplexity**: 
  - 入力欄: `#ask-input`（Lexical editor固有のID）
  - 送信ボタン: `button[aria-label='送信']`

### 🔄 各画面への水平展開
- 「AI一括取得画面」および「確認モードのスキャン詳細」の両方で、上記の修正を同期。どちらの導線からでも一貫して正しいAIが開き、プロンプトが自動入力されるように改善。

## 変更点5：Perplexity (Lexical エディタ) への完全対応

**対象ファイル**: `WebViewJsHelper.kt`, `自動貼り付け取り込みの仕組み.md`

### 🎯 修正: Perplexity 固有の要素特定
- **背景**: Perplexity の入力欄は `id="ask-input"` を持つ Lexical エディタであり、標準的なセレクタでは特定が不安定な場合があった。
- **対策**: `#ask-input` および `data-lexical-editor="true"` を最優先セレクタとして設定し、確実にターゲットを補足するように修正。

### ⚡ 強化: Lexical 互換のインジェクション
- **問題**: モダンなエディタフレームワーク（Lexical）では、単純なテキスト代入では「未入力」と判定され、送信ボタンが有効にならない。
- **対策**: 
  - `InputEvent` を生成し、`beforeinput` イベントを `insertText` タイプで明示的に発火させる処理を追加。
  - これによりエディタ内部のデータモデルが強制的に更新され、Web サイト側に「正しく入力された」と認識させることに成功。

### 🖱️ 修正: 送信ボタンのクリック精度向上
- **対策**: ボタン内部に複雑な SVG 構造がある場合でも、`aria-label="送信"` を持つ `button` 要素を確実に特定し、クリックイベントを伝達するようにセレクタチェーンを調整。

## 変更点6：WebViewスクロール対応・JS注入安定化・タイムアウト改善

**対象ファイル**: `WebViewJsHelper.kt`, `AiFetchViewModel.kt`  
**日付**: 2026-04-06

### 📜 修正: 設定画面AIサイトログインWebViewのスクロール対応
- **問題**: 設定画面（`SettingsScreen.kt`）のAIサイトログインWebViewが、外側の `verticalScroll` と競合しスクロールできなかった。
- **対策**:
  - WebView側で `isVerticalScrollBarEnabled = true`・`isHorizontalScrollBarEnabled = true` を有効化。
  - `settings.setSupportZoom(true)` / `settings.builtInZoomControls = true` を設定し、ピンチズームも有効化。
  - Compose側でWebViewを `weight(1f)` + `fillMaxSize()` で指定し、固定高さによる制約を解除。
  - WebViewを包む `Box` に `Modifier.nestedScroll(connection)` は付与せず、WebView内スクロールをネイティブに委譲。

### 🔧 修正: プロンプト注入スクリプトの刷新（Lexical/ProseMirror完全対応）

| 変更前 | 変更後 |
|---|---|
| `execCommand('insertText')` のみ（Chrome WebViewで動作不安定） | `DataTransfer` + `ClipboardEvent('paste')` でLexicalに直接ペースト |
| `el.innerText = prompt` で直接上書き（Reactの仮想DOMを無視） | pasteイベント後に `innerText` が空の場合のみ fallback として代入 |
| `for...of` ループ | `for (var i = 0; ...)` に変更（旧WebView互換性向上） |

**注入フロー（新）**:
1. `selectAll` → `delete` でクリア
2. `DataTransfer` に text をセットして `paste` イベント発火（Lexicalが最も確実に反応）
3. `innerText` が空なら直接代入（フォールバック）
4. `beforeinput` → `input` → `change` イベントを順に発火

### 🎯 修正: 送信ボタンセレクタ刷新・MouseEvent対応

- **Gemini**: `button[jsaction*='send']`、`button[data-mat-icon-name='send']` を追加
- **Perplexity**: `button[data-testid='send-button']` を追加、SVG子要素でのクリックを廃止
- `mousedown` + `mouseup` + `click` の3段階発火でフレームワークのクリックハンドラを確実に起動
- ボタンが見つからない場合は `Enter` キーイベントのフォールバックを追加

### 🔍 修正: レスポンス抽出セレクタ刷新

- **Gemini**: `model-response`（現行カスタム要素）、`div[data-message-author-role='model']` を追加
- **Perplexity**: `div[data-testid='answer-content'] .prose` を追加
- 取得テキストが **5文字以下なら無視**（生成途中の断片を拾わないよう）

### ⏱️ 修正: `evaluateJsSync` のハングアップ解消

```kotlin
private suspend fun evaluateJsSync(script: String): String? {
    val wv = webView ?: return null  // null なら即 return
    return try {
        withTimeout(10_000L) {       // 10秒タイムアウト
            suspendCancellableCoroutine { continuation ->
                wv.post {
                    wv.evaluateJavascript(script) { result ->
                        if (continuation.isActive) {
                            continuation.resumeWith(Result.success(result))
                        }
                    }
                }
            }
        }
    } catch (e: TimeoutCancellationException) {
        null
    }
}
```

### 🐛 修正: JS戻り値の判定ロジック（JSON文字列ラップ対応）

- `evaluateJavascript` は JavaScript の `true` を JSON文字列 `"true"`（引用符付き）として返す仕様のため、従来の `== "true"` 判定が常に失敗していた。
- **対策**: `injectResult == "true" || injectResult == "\"true\""` の両パターンに対応。

### 📊 その他改善

- ポーリング回数を **20回 → 30回**（最大30秒待機）に延長
- レスポンスのアンエスケープ処理に `\t`・`\\` の処理を追加

## 変更点8：AI自動取得機能の抜本的強化（GikobunAI方式への移行）

**日付**: 2026-04-06
**対象ファイル**: `AiWebViewWrapper.kt`, `AiWebViewInteractor.kt`, `AiFetchViewModel.kt`, `ScanViewModel.kt`

### ■ ステップ1：WebViewの基礎設定更新とUserAgentの固定化

- **変更内容**: `AiWebViewWrapper.kt` において、UserAgentを完全なブラウザ版（Chrome 120系）に固定。
  - **以前**: `userAgentString.replace("; wv", "")`（モバイルWebView識別子の削除のみ）
  - **以後**: `"Mozilla/5.0 (Linux; Android 13; Pixel 7) " + "AppleWebKit/537.36 (KHTML, like Gecko) " + "Chrome/120.0.0.0 Mobile Safari/537.36"` に固定。
- **目的**: 
  - AIサイト（特にPerplexityやGemini）における「サポートされていないブラウザ」警告の回避。
  - モバイル版UI特有の制約（入力欄の挙動やボタンの配置）を排除し、デスクトップ/標準ブラウザに近い安定したDOM操作を可能にする。
- **検証**: ビルド（assembleDebug）成功。

### ■ ステップ2：AiWebViewInteractor.kt の刷新（GikobunAI方式）

- **変更内容**: `AiWebViewInteractor.kt` を GikobunAI で実績のある「統合JS実行方式」に全面刷新。
  - **JavascriptInterface (`JanBridge`)**: JS側からの結果・エラー・ステータス通知をネイティブで受け取る仕組みを導入。
  - **統合JS (`buildAllInOneJs`)**: 1回の `evaluateJavascript` 実行で以下の全工程をJS側のタイマー制御（`setTimeout`/`setInterval`）で完結させる。
    - 入力欄のDOM出現待ち（最大20秒）
    - `document.execCommand('insertText')` によるプロンプト注入（React/Lexical等への確実な反映）
    - 送信ボタンの自動クリック（またはEnterキー送信）
    - レスポンスの安定判定（3秒間テキスト長に変化がなければ完了とみなす）
  - **非同期待機**: Kotlin側では `suspendCancellableCoroutine` を使用し、JS側からのコールバックがあるまで中断・待機する設計に変更。
- **効果**: Kotlin↔JS間の頻繁な通信を排除し、ネットワーク遅延やSPAの非同期レンダリングに左右されない極めて堅牢な自動操作を実現。
- **検証**: ビルド（assembleDebug）成功。

### ■ ステップ3・4：各 ViewModel の検証

- **内容**: `AiFetchViewModel.kt` および `ScanViewModel.kt` を確認。
- **結果**: 
  - 両 ViewModel ともに既に `AiWebViewInteractor` をプロパティとして保持し、`executeFullFlow` を呼び出す構造になっているため、ロジック自体の変更は不要であることを確認。
  - Interactor の内部実装が刷新されたことにより、ViewModel 側のコードを変更することなく、より高性能な自動操作機能が自動的に適用される状態となった。

## 変更点10：デバッグログ分析に基づくJS構文エラーの修正と堅牢化

**日付**: 2026-04-06
**対象ファイル**: `AiWebViewWrapper.kt`, `AiWebViewInteractor.kt`

### 🐛 修正: セレクタのエスケープ不備によるJS構文エラーの解消
- **問題**: ログ分析の結果、Kotlin側で「Executing JS...」と記録されているにもかかわらず、JS側のログが一切出力されていないことが判明。原因は `escapeSelectorForJs` の置換順序の誤りによる構文エラーだった。
  - **以前**: `'` → `\'` → `\\'` と変換され、JS上ではシングルクォートが文字列の終端と判定され、以降のコードが構文エラーになっていた。
  - **対策**: 置換順序を修正（`\` を先に置換し、その後に `'` を置換）することで、シングルクォートを含むセレクタ（例: `div[contenteditable='true']`）が正しくJSに渡されるようにした。

### 🛡️ 強化: JS実行時エラーの捕捉と報告 (try-catch)
- **対策**: 注入する全JSコードを `try-catch` ブロックで囲み、万一実行時エラーが発生した際も `JanBridge.onError` を通じて Kotlin 側にエラー内容（メッセージ）を即座に報告するように強化。
- **改善**: スクリプトの冒頭に `console.log` を配置し、実行開始が確実に Logcat (`chromium` タグ) に記録されるようにした。

### 🔄 改善: WebViewブリッジ登録の確実な再試行
- **対策**: `AiWebViewWrapper.kt` の `onPageFinished` において、`onWebViewCreated` (ブリッジ登録) を再度呼び出すようにし、SPAサイトでのページ遷移後もブリッジが確実に利用可能な状態を維持するようにした。

### ✅ 検証結果
- `gradlew assembleDebug` によるビルド成功を確認。構文エラーが解消され、JSの実行開始が担保された状態。

## 変更点11：一括取得フローの安定化（インデックス依存の排除とJANベース追跡）

**日付**: 2026-04-06
**対象ファイル**: `AiFetchViewModel.kt`

### 🛠️ 修正: 非同期リスト更新に伴うインデックス不整合の解消
- **問題**: AIからの取得成功後、承認ボタンを押してもデータが更新されず、次へ進まない場合がある。原因は、非同期で更新される `unfetchedProducts` リストと `currentIndex` の不整合だった。
  - リストが更新されて件数が減った際、古いインデックスを参照して `getOrNull` が失敗し、処理が中断されていた。
  - 処理が中断されると、次の商品へ進むためのシグナル (`_proceedSignal`) が発行されず、ループがフリーズしていた。
- **対策**: 
  - **JANコードベースの追跡**: `onAcceptResult` 内で、インデックスではなく `result.jan_code` を使用してDBから商品を直接取得して更新するように変更。
  - **ループ制御の適正化**: 
    - 成功・承認時は商品がリストから消えるため、`currentIndex` を増やさない（次の商品が同じインデックスに繰り上がるため）。
    - 失敗・却下・エラー時のみ `currentIndex` を増やして次の商品へ進めるようにロジックを整理。
  - **フリーズ防止**: どのような中断経路でもシグナルが適切に発行（またはループが終了）されるようガードを強化。

### ✅ 検証結果
- `gradlew assembleDebug` によるビルド成功を確認。リストの動的な増減に左右されない、堅牢な取得フローになった。
