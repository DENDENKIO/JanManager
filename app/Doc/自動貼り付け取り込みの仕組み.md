# 自動貼り付け・取り込みの仕組み（設計書）

本ドキュメントでは、JanManager アプリケーションにおける WebView への自動入力と情報抽出の「完成されたロジック」について解説します。

## 1. 安定稼働のための 5 段階プロセス

動作が安定している「スキャン詳細画面」のロジックに基づき、以下のステップで実行します。

1.  **Pre-check (ページ遷移待機)**
    *   WebView の現在の URL を監視し、ターゲット（Gemini または Perplexity）のドメインが含まれるまで最大 10 秒間待機します。
    *   これにより、遷移途中の不完全な状態でスクリプトが実行されるのを防ぎます。

2.  **Injection Loop (リトライ付き注入)**
    *   入力欄の特定とテキストの流し込みを、最大 3 回までリトライします。
    *   各試行間に `delay(2000)` を置くことで、重いページの描画待ちに対応します。

3.  **Verification (成否判定)**
    *   JavaScript 側から実行結果を `true/false` の文字列として返します。
    *   Kotlin 側でこれを受け取り、成功が確定するまで次の「送信ステップ」へ進まないように制御します。

4.  **Action (送信実行)**
    *   貼り付け（注入）の成功を確認した直後（約 0.8 秒後）に、送信ボタンのクリックを実行します。
    *   送信ボタン自体もフォールバックチェーン（チェーン内の複数の候補を順次試行）で特定します。

5.  **Polling & Parsing (回答抽出)**
    *   AI の回答エリアを 1 秒ごとに監視し、内容の更新が止まるまで（または JSON が完成するまで）ループします。
    *   取得した文字列から正規表現で JSON を抽出し、JAN コードの整合性バリデーションを行います。

---

## 2. 実装のキモ：リトライと検証 (Retry & Verify)

単に JS を投げるだけでは、ネットワーク速度や端末負荷の個体差により「要素が見つからない（貼り付け失敗）」が高頻度で発生します。

### A. Kotlin と JS の同期連携
`evaluateJavascript` のコールバックを利用した `evaluateJsSync` 通信を行い、JS 側の `return true;` を待機します。この「確実な手応え」を確認してから処理を進めるのが、安定動作の最大のポイントです。

### B. 強力なインジェクション（注入）スクリプト
モダンな AI サイト（React/Lexical 等）に対応するため、以下のステップを 1 回の JS 実行で完結させます。

1.  **特定**: フォールバックチェーン（マニュアル設定 ＞ ARIA ラベル ＞ 専用クラス ＞ ID ＞ タグ名）で要素を取得。特に `id="ask-input"` や `data-lexical-editor="true"` を優先します。
2.  **挿入**: `execCommand('insertText')` を第一優先とし、失敗時に `innerText` または `value` 代入へ切り替え。
3.  **高度な通知**: `InputEvent` を生成し、`beforeinput` イベントを `inputType: 'insertText'` と共に発火させます。これにより、Lexical などのエディタモデルを確実に更新させます。
4.  **イベント発火**: `input`, `change`, `blur`, `keyup` の 4 つのイベントを連鎖的に発生させ、サイト側の「送信ボタン有効化ロジック」を起動させます。

### C. 送信ボタンの深層探索
ボタン内部に複雑な SVG 構造（Perplexity 等）がある場合でも、ARIA ラベル (`aria-label="送信"`) を基点に検索を行い、確実なクリックイベントを伝達します。

---

## 3. 他機能への水平展開の指針

新しい Web サイトや自動化機能を実装する場合も、この「Pre-check → Retry & Verify → Action」の設計パターンを崩さずに適用することで、不安定な要素を極限まで排除できます。
-   **セレクタの管理**: `WebViewJsHelper` にフォールバックリストを集約します。
-   **ステータス表示**: ユーザーに現在の試行回数（1/3 等）を提示し、安心感を与えます。
